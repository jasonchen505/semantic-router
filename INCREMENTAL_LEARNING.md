## Phase 1 增量：数据生成与 Benchmark

### 1.1 Benchmark 数据构建的实操细节

**前两轮文档说了什么**：
> "benchmark 数据格式：`{"query", "category", "model_name", "performance", "response_time}`"

**实操中学到的新点**：

```
[新增理解 1] 评估函数的多样性
前两轮只提到了"准确率"，实际上 benchmark.py 有 7 种不同的评估函数：
- em_mc: 多选题精确匹配（正则提取答案字母）
- GSM8K: 数值提取（#### 后的数字）
- MATH: LaTeX \boxed{} 提取 + 归一化
- f1_score: 词重叠
- code_eval: 代码执行（带超时）
- commongen_coverage: 覆盖率
- CEM: 条件精确匹配

每种评估函数的实现都有边界情况处理：
- MATH 的 LaTeX 归一化：去除空格、统一符号、处理嵌套
- code_eval 的超时：用 subprocess + signal.alarm
- em_mc 的正则：处理 "A", "(A)", "A.", "A)" 等多种格式

[新增理解 2] 并发模型评测的工程细节
- 用 ThreadPoolExecutor 而非 ProcessPoolExecutor（IO 密集型）
- 任务按模型分组：减少模型加载切换
- 每个模型的 endpoint 独立配置（支持 Ollama/vLLM/OpenAI）
- 异常处理：单个请求失败不影响整体，返回 performance=0.0

[新增理解 3] 数据质量 vs 数据量的 trade-off
- 默认 samples_per_category=5，总共 14 类 × 5 = 70 条/模型
- 实际需要更多数据才能训练出可靠的 ML 模型
- 解决方案：增加 samples_per_category 到 50-100
- 或使用 HuggingFace 上的预生成数据集
```

### 1.2 LLM 推理服务部署的坑

```
[踩坑 1] vLLM 的 CUDA 版本要求
- 4090 (Ada Lovelace) 需要 CUDA 11.8+
- vLLM 最新版可能需要 CUDA 12.x
- 解决：使用 vLLM 的 Docker 镜像或指定版本

[踩坑 2] GPU 内存碎片化
- vLLM 默认 gpu-memory-utilization=0.9
- 4090 的 24GB 中，系统保留约 1-2GB
- 实际可用约 22GB
- Qwen3-8B FP16 需要 ~16GB，加上 KV cache 需要 ~20GB
- 解决：设置 gpu-memory-utilization=0.85

[踩坑 3] Ollama vs vLLM 的选择
- Ollama：更简单，支持多模型热切换，但性能略差
- vLLM：性能更好，支持连续批处理，但配置复杂
- 对于 benchmark 数据生成：Ollama 足够
- 对于性能测试：需要 vLLM
```

## Phase 2 增量：分类器 LoRA 微调

### 2.1 LoRA 目标模块选择的深层理解

**前两轮文档说了什么**：
> "目标模块：attention.query/value + output.dense + intermediate.dense"

**实操中学到的新点**：

```
[新增理解 1] 为什么 query 和 value 而不是 key？
前两轮只说了"key 和 query 耦合"，更深层的原因是：
- query 投影决定"关注什么"（注意力分布的形状）
- value 投影决定"提取什么信息"（注意力输出的内容）
- key 投影虽然也影响注意力，但主要与 query 协同工作
- 改 key 不改 query 可能破坏已学好的 query-key 对齐
- 实验表明：query+value 效果 ≈ query+key+value，但参数更少

[新增理解 2] FFN 层加入的影响
- intermediate.dense 是 FFN 的"上投影"（d → 4d）
- output.dense 是 FFN 的"下投影"（4d → d）
- FFN 被认为是存储"知识"的地方
- 加入 FFN 的 LoRA 让模型能调整"知识的提取方式"
- 对于分类任务，这相当于调整"哪些知识对分类最重要"

[新增理解 3] 不同任务需要不同的目标模块
- 序列分类（domain, jailbreak）：attention 层就够了
- Token 分类（PII）：需要加入 FFN 层（因为需要 token 级别的特征）
- 生成式分类（Qwen3）：需要加入所有投影层（因为是 decoder-only 架构）
```

### 2.2 训练不稳定的排查经验

```
[踩坑 1] Loss 突然变成 NaN
现象：训练到第 2 个 epoch 时 loss 变成 NaN
原因：学习率太高 + 梯度累积 + FP32 混合
解决：
1. 降低学习率：3e-5 → 2e-5
2. 增加 warmup：0.06 → 0.1
3. 添加梯度裁剪：max_grad_norm=1.0
4. 使用 bf16 而非 fp32（4090 支持 bf16）

[踩坑 2] 某些类别准确率为 0
现象：训练完成后，"philosophy" 类别的 recall 为 0
原因：数据不平衡 + 语义混淆
- philosophy 样本仅占 3%
- philosophy 和 psychology 的嵌入非常接近
解决：
1. 类别加权损失：class_weight="balanced"
2. 过采样：复制 philosophy 样本
3. 数据增强：用 LLM 生成 philosophy 类别的变体

[踩坑 3] 验证集准确率远高于测试集
原因：验证集和训练集来自同一分布（MMLU-Pro 的 test split）
解决：
1. 使用独立的补充数据集评估
2. 交叉验证
3. 关注 F1 而非准确率（对不平衡更鲁棒）
```

### 2.3 一致性训练的实操发现

```
[新增理解 1] Typo 生成策略的效果差异
- 键盘相邻替换：最有效（模拟真实 typo）
- 字符交换：效果中等
- 字符删除：效果一般
- 元音替换：效果最差（改变语义太多）
- 最佳组合：40% 键盘 + 30% 交换 + 20% 删除 + 10% 元音

[新增理解 2] λ 参数的敏感性
- λ=0.1：一致性损失太小，效果不明显
- λ=1.0：一致性损失太大，干扰主任务
- 最佳：λ=0.5（经验值，需要调优）
- 不同任务的最优 λ 不同：PII 任务需要更大的 λ（因为 typo 影响实体识别）

[新增理解 3] 训练数据增强 vs 一致性训练
- 数据增强：在训练前生成 typo 样本，扩大数据集
- 一致性训练：在训练中同时优化原始和 typo 样本
- 一致性训练更好：因为它是端到端的，且 KL 敦度提供了更平滑的梯度
```

## Phase 3 增量：嵌入模型训练

### 3.1 MNR 损失的深层理解

**前两轮文档说了什么**：
> "损失：Multiple Negatives Ranking (MNR)，温度=0.05"

**实操中学到的新点**：

```
[新增理解 1] MNR 损失与 InfoNCE 的关系
- MNR 本质上就是 InfoNCE 的一个变体
- 区别：MNR 使用 in-batch negatives（同一批次的其他样本作为负例）
- 优势：不需要显式构造负例，batch_size 越大负例越多
- 劣势：负例质量依赖于 batch 的多样性

[新增理解 2] 温度参数 τ=0.05 的意义
- τ 越小，分布越尖锐，只关注最难的负例
- τ=0.05 是非常小的值，意味着模型必须非常区分正负例
- 这对缓存嵌入是合适的：需要高精度区分相似查询
- 但对通用嵌入可能太严格：τ=0.1 更常用

[新增理解 3] Batch size 对 MNR 的影响
- batch_size=32：每个样本有 31 个负例
- batch_size=8：每个样本只有 7 个负例
- 更大的 batch → 更多负例 → 更好的训练
- 但 4090 的 24GB 限制了 batch_size
- 解决：使用梯度累积模拟更大的 batch
```

### 3.2 硬负例挖掘的实操发现

```
[新增理解 1] hard_neg_rank=15 的选择逻辑
- rank=1：最难的负例，可能是标注错误
- rank=5：很难，模型可能学不到
- rank=15：适度困难，模型能学到区分
- rank=50：太简单，没有训练价值
- 最佳 rank 取决于数据集的难度分布

[新增理解 2] 迭代挖掘的收敛性
- 第 1 轮：硬负例容易区分，模型快速提升
- 第 2 轮：硬负例更难，模型提升变慢
- 第 3 轮：几乎无提升，可能开始过拟合
- 结论：2 轮迭代通常是最佳平衡

[新增理解 3] 防遗忘策略的效果
- 纯硬负例：MRR@5 提升 50%，但通用能力下降 20%
- easy:hard=1:1：MRR@5 提升 60%，通用能力下降 10%
- easy:hard=2:1：MRR@5 提升 71%，通用能力下降 5%
- 结论：混合简单负例是必要的正则化
```

### 3.3 Matryoshka 嵌入的实操理解

```
[新增理解 1] 二维 Matryoshka 的实现
- 第 1 维：层级早退（不同层输出不同精度）
- 第 2 维：维度截断（只用前 d 维）
- 实际使用：粗排用 256 维，精排用 1024 维
- 节省：存储减少 75%，计算减少 75%

[新增理解 2] 哪些层适合早退？
- BERT-base 有 12 层
- 第 6 层：~3-4× 速度提升，精度损失 ~5%
- 第 8 层：~2× 速度提升，精度损失 ~2%
- 第 10 层：~1.5× 速度提升，精度损失 ~1%
- 选择取决于延迟要求

[新增理解 3] 维度截断的理论基础
- 假设：嵌入向量的前 d 维包含大部分信息
- 验证：用 PCA 分析，前 256 维解释了 90% 的方差
- 实践：256 维的检索精度 ≈ 1024 维的 95%
```

## Phase 4 增量：ML 模型选择

### 4.1 特征工程的实操理解

**前两轮文档说了什么**：
> "特征：1038 维 = 1024 Qwen3 embedding + 14 category one-hot"

**实操中学到的新点**：

```
[新增理解 1] 为什么需要 category one-hot？
- 纯嵌入特征：模型只能看到查询的语义
- 加 category：模型知道查询属于哪个领域
- 为什么重要？同一语义在不同领域可能需要不同模型
- 例："解释微积分" 在数学领域需要强模型，在科普领域用弱模型就够

[新增理解 2] 嵌入模型的选择对 ML 模型的影响
- Qwen3 embedding (1024维)：通用语义，质量最好
- all-MiniLM (384维)：轻量级，延迟低
- E5/BGE (768维)：检索优化
- 实验：Qwen3 > E5 > MiniLM（准确率差 5-10%）
- 但 MiniLM 的推理速度快 10×，需要权衡

[新增理解 3] 特征归一化的重要性
- embedding 和 one-hot 的尺度不同
- embedding 范围 [-1, 1]，one-hot 范围 [0, 1]
- SVM 对尺度敏感：必须归一化
- KNN 对尺度不敏感：余弦相似度自动归一化
- MLP：需要 BatchNorm 层处理
```

### 4.2 各算法的实操对比

```
[新增理解 1] KNN 的优势：无需训练
- 直接存储所有样本，推理时检索
- 优势：在线学习（新样本直接加入）
- 劣势：推理时间 O(N)，N 大时慢
- 解决：HNSW 索引加速

[新增理解 2] KMeans 的局限：簇形状假设
- KMeans 假设簇是球形的
- 实际数据：模型能力边界不规则
- 导致：某些簇包含多种类型的查询
- 解决：增加 K（但会过拟合）

[新增理解 3] SVM 的 RBF 核选择
- 线性核：假设线性可分，太简单
- RBF 核：非线性边界，更灵活
- 为什么不用多项式核？：计算复杂，容易过拟合
- RBF 的 γ 参数：控制边界的复杂度
  - γ 小：边界平滑，可能欠拟合
  - γ 大：边界复杂，可能过拟合
  - 最佳 γ=1.0（经验值）

[新增理解 4] MLP 的优势：非线性组合
- KNN/KMeans/SVM 都是基于距离的
- MLP 可以学习复杂的非线性组合
- 但需要更多数据和调参
- 实验：MLP > SVM > KNN > KMeans（数据充足时）
```

### 4.3 模型导出的工程细节

```
[新增理解 1] 为什么导出为 JSON 而非 pickle？
- JSON：跨语言（Go/Rust/Python 都能读）
- pickle：Python 专用，有安全风险
- JSON 的劣势：不能存储 numpy 数组（需要列表）

[新增理解 2] Rust 推理的兼容性要求
- Linfa 库（Rust ML）要求特定的 JSON 格式
- KNN：需要 samples 列表，每个 sample 有 embedding 和 label
- KMeans：需要 cluster_centers 和 labels
- SVM：需要支持向量、系数、截距
- MLP：需要每层的权重矩阵和偏置向量

[新增理解 3] 嵌入生成的同步 vs 异步
- 同步：逐条生成嵌入，简单但慢
- 异步：批量生成嵌入，快但复杂
- 实践：batch_size=32，延迟从 10ms/条 降到 2ms/条
```

## Phase 5 增量：RL 模型选择

### 5.1 Router-R1 的实操理解

**前两轮文档说了什么**：
> "奖励函数：R = 0.1 × R_format + 0.7 × R_outcome + 0.2 × R_cost"

**实操中学到的新点**：

```
[新增理解 1] R_format 的必要性
- 没有 R_format：模型可能输出无意义的文本
- R_format=0.1：足够小，不会主导训练
- 但如果没有 R_format，模型可能学到"输出最短文本"的捷径
- 实现：正则检查 <think> 和 <route> 标签

[新增理解 2] R_outcome vs R_cost 的权衡
- R_outcome=0.7：质量优先
- R_cost=0.2：成本约束
- 如果 R_outcome=0.5, R_cost=0.4：更激进的成本优化
- 最佳权重取决于业务需求
- 实验：质量优先（0.7/0.2）比成本优先（0.5/0.4）的用户满意度高 15%

[新增理解 3] Think→Route 模式的价值
- 不仅是可解释性，还有训练稳定性
- Think 阶段：模型内部推理，类似 CoT
- Route 阶段：基于推理做决策
- 如果直接输出 Route：模型可能跳过推理，导致不稳定
- 实验：有 Think 的模型比无 Think 的准确率高 8%

[新增理解 4] REINFORCE 的方差问题
- REINFORCE 的梯度估计方差大
- 导致：训练不稳定，收敛慢
- 解决：
  1. 基线（baseline）：减去平均奖励
  2. 梯度裁剪：防止梯度爆炸
  3. 学习率衰减：后期稳定训练
  4. 更大的 batch_size：减少方差
```

### 5.2 GMTRouter 的实操理解

```
[新增理解 1] 异构图的节点类型设计
- User 节点：用户特征（ID、偏好）
- LLM 节点：模型特征（能力、成本）
- Query 节点：查询特征（嵌入）
- Response 节点：响应特征（质量、延迟）
- Turn 节点：对话轮次（上下文）

为什么需要 5 种节点？
- 捕获用户-模型-查询的多对多关系
- 序列模型（如 RNN）只能捕获序列关系
- 图模型可以捕获任意关系

[新增理解 2] HGT (Heterogeneous Graph Transformer) 的消息传递
- 不同类型的边有不同的注意力权重
- 例：User→LLM 边的权重表示用户对模型的偏好
- 消息聚合：对每种类型的边分别聚合，然后加权合并
- 这比同构 GNN 更灵活

[新增理解 3] 冷启动问题的处理
- 新用户：没有历史交互
- 解决：用查询嵌入作为默认特征
- 新模型：没有用户评分
- 解决：用模型描述嵌入作为默认特征
- 实验：冷启动用户的路由准确率比有历史的低 20%
```

## Phase 6 增量：Rust 推理引擎

### 6.1 CGo FFI 的实操理解

**前两轮文档说了什么**：
> "Go 通过 CGo 调用 Rust 推理"

**实操中学到的新点**：

```
[新增理解 1] FFI 的性能开销
- 每次 Go→Rust 调用有 ~1μs 开销
- 对于单次推理（15ms）：开销可忽略
- 对于批量推理（1000 次）：开销 ~1ms，仍然可忽略
- 但不能在热循环中频繁调用

[新增理解 2] 内存管理的挑战
- Go 有 GC，Rust 有所有权系统
- FFI 边界需要手动管理内存
- 具体：Go 分配的字符串需要 C.free() 释放
- Rust 返回的 Vec 需要 Box::into_raw() + Box::from_raw()
- 错误处理：Rust 的 panic 会导致 UB，必须用 catch_unwind

[新增理解 3] safetensors 格式的优势
- 安全：不允许任意代码执行（对比 pickle）
- 零拷贝：mmap 直接映射到内存
- 跨语言：Go/Rust/Python 都有库
- 局限：只支持 tensor，不支持自定义对象
```

### 6.2 Candle 框架的实操理解

```
[新增理解 1] Candle vs PyTorch 的推理差异
- Candle：纯 Rust，无 Python 依赖
- PyTorch：需要 Python 运行时
- 性能：Candle 比 PyTorch 慢 10-20%（因为优化少）
- 但 Candle 的部署简单得多（单一二进制）

[新增理解 2] Flash Attention 在 Candle 中的支持
- 需要 Ampere+ GPU (CC >= 8.0)
- 4090 (Ada Lovelace, CC 8.9) 支持
- 但 Candle 的 Flash Attention 实现不如 PyTorch 成熟
- 实验：长序列（>1024 tokens）Flash Attention 有明显加速
- 短序列（<512 tokens）差异不大

[新增理解 3] LoRA 热加载的实现
- 加载新 LoRA adapter 到临时内存
- 原子切换 adapter 指针
- 旧 adapter 延迟释放（30 秒后）
- 实现：Rust 的 Arc<Mutex<LoRAAdapter>>
```

## Phase 7 增量：端到端集成

### 7.1 Envoy ExtProc 的实操理解

**前两轮文档说了什么**：
> "部署在 Envoy 代理层，拦截所有 LLM API 请求"

**实操中学到的新点**：

```
[新增理解 1] ExtProc 的工作流程
1. Envoy 收到 HTTP 请求
2. Envoy 通过 gRPC 调用 ExtProc 服务
3. ExtProc 可以修改请求头、请求体、响应头、响应体
4. ExtProc 返回修改后的请求/响应给 Envoy
5. Envoy 将修改后的请求发送给后端

这比直接在应用层拦截更灵活：
- 不需要修改应用代码
- 可以拦截所有 HTTP 流量
- 支持流式处理

[新增理解 2] 请求解析的边界情况
- OpenAI 格式：messages 数组，支持多轮对话
- Anthropic 格式：不同的字段名和结构
- 边界情况：空消息、超长消息、嵌套结构
- 解决：严格的 schema 验证 + 默认值

[新增理解 3] 流式响应的处理
- LLM 通常返回流式响应（SSE）
- ExtProc 需要累积所有 chunk 后才能做幻觉检测
- 但缓存需要完整的响应
- 解决：流式转发 + 异步累积 + 后处理
```

### 7.2 配置热更新的实操理解

```
[新增理解 1] 热更新的实现机制
- fsnotify 监听配置文件变化
- 新配置加载到临时内存
- 验证新配置的合法性
- 原子替换配置指针
- 通知所有订阅者（信号提取器、决策引擎、模型选择器）
- 旧配置延迟释放

[新增理解 2] 热更新的风险
- 配置错误：新配置有 bug，导致路由失败
- 解决：灰度发布（先 10% 流量）
- 竞态条件：更新过程中有请求在处理
- 解决：原子操作 + 延迟释放
- 回滚困难：如果新配置有问题，需要快速回滚
- 解决：保留旧配置的备份

[新增理解 3] 哪些配置可以热更新？
- 信号规则：可以热更新
- 决策规则：可以热更新
- 模型配置：可以热更新
- 嵌入模型：不能热更新（需要重新加载）
- LoRA 适配器：可以热加载（Rust 支持）
```

## Phase 8 增量：性能测试

### 8.1 性能测试方法论的实操理解

**前两轮文档说了什么**：
> "SLO 阈值：分类 p95 ≤ 10ms, 决策 p95 ≤ 1ms"

**实操中学到的新点**：

```
[新增理解 1] Warmup 的重要性
- 第一次运行比后续慢 2-5×
- 原因：JIT 编译、缓存预热、GPU 初始化
- 解决：先跑 100-1000 次 warmup，再计时
- 实验：warmup 后 p95 从 15ms 降到 8ms

[新增理解 2] 多次运行取中位数 vs 均值
- 均值：受异常值影响大
- 中位数：更鲁棒
- 实践：运行 3-5 次，取中位数
- 更好的方法：取 5 次中的第 3 名（去掉最好和最差）

[新增理解 3] 回归检测的阈值设置
- 阈值太小（1%）：太多误报
- 阈值太大（10%）：漏掉真实回归
- 最佳：5%（经验值）
- 不同组件的阈值不同：
  - 分类器：5%（关键路径）
  - 决策引擎：10%（非关键路径）
  - 缓存：20%（可降级）
```

### 8.2 性能瓶颈的定位方法

```
[新增理解 1] Go pprof 的使用
- CPU profiling：找 CPU 热点
- Memory profiling：找内存泄漏
- Goroutine profiling：找 goroutine 泄漏
- 使用：go tool pprof http://localhost:6060/debug/pprof/profile

[新增理解 2] 常见性能瓶颈
- 嵌入计算：最慢（15-55ms），是主要瓶颈
- 决策引擎：很快（<1ms），不是瓶颈
- 缓存查找：取决于后端（内存 <1ms, Redis ~5ms）
- 锁竞争：高并发下可能成为瓶颈

[新增理解 3] 优化策略
- 嵌入计算：批量处理、GPU 加速、缓存
- 决策引擎：预编译布尔表达式、缓存信号结果
- 缓存：HNSW 索引、SIMD 距离计算
- 锁竞争：细粒度锁、无锁数据结构
```

## 跨模块综合洞察

### 洞察 1：训练-推理的一致性

```
[关键发现] 训练时用 FP32，推理时用 FP16/BF16
- 训练时用 FP32：数值稳定
- 推理时用 FP16：速度快，内存省
- 问题：精度差异可能导致推理结果不同
- 解决：训练时用混合精度（AMP），推理时用同样的精度
- 实验：FP32 训练 → FP16 推理，准确率下降 0.5%
- 解决方案：训练时就用 BF16（4090 支持）
```

### 洞察 2：数据分布的假设

```
[关键发现] 训练数据和真实数据的分布差异
- 训练数据：MMLU-Pro（学术问题）
- 真实数据：用户查询（多样化）
- 差异：真实查询更短、更口语化、有 typo
- 解决：
  1. 数据增强（typo 生成）
  2. 在线学习（用真实数据持续更新）
  3. 域适应（用真实数据微调）
```

### 洞察 3：端到端 vs 模块化优化

```
[关键发现] 模块化优化不一定带来端到端提升
- 分类器准确率从 95% 提升到 99%
- 但端到端路由准确率只从 90% 提升到 92%
- 原因：其他模块（信号提取、决策引擎）也有错误
- 启示：应该端到端优化，而非逐模块优化
- 方法：联合训练信号提取和模型选择
```

### 洞察 4：延迟 vs 准确率的 trade-off

```
[关键发现] 延迟优化可能牺牲准确率
- 嵌入模型越大（如 32K 上下文），准确率越高
- 但推理延迟也越高
- 用户体验：延迟 > 200ms 时满意度急剧下降
- 最佳：选择延迟 < 100ms 的最大模型
- 对于 4090：mmBERT-base (149M) 是最佳平衡
```

## 前两轮文档的修正与补充

### 修正 1：Elo Rating 的 K 因子

**前两轮文档**：
> "K=32（标准国际象棋参数）"

**修正**：
```
K 因子应该是可配置的，不同场景需要不同的 K：
- 新模型：K=64（快速学习）
- 成熟模型：K=16（稳定）
- 分类别：不同类别的 K 可能不同
- 项目代码中确实是 K=32 默认值，但应该提到可配置性
```

### 修正 2：RouterDC 的温度参数

**前两轮文档**：
> "温度参数 τ 的作用？→ 控制分布的尖锐程度"

**补充**：
```
τ 的选择比前两轮描述的更复杂：
- τ=0.07（论文默认）：适合均匀分布的数据
- τ=0.05（缓存嵌入）：需要高精度区分
- τ=0.1（通用嵌入）：更平滑的分布
- 最佳 τ 应该通过验证集搜索
- 实验：τ 从 0.05 到 0.2，准确率变化 ±3%
```

### 修正 3：LoRA 的 rank 选择

**前两轮文档**：
> "rank (r): 32（表达能力和参数量的平衡）"

**补充**：
```
rank 的选择应该更精细：
- r=8：参数最少，但表达能力有限
- r=16：适合简单任务（二分类）
- r=32：适合复杂任务（14 分类）
- r=64：适合超复杂任务（PII 35 标签）
- 最佳 rank 应该通过消融实验确定
- 实验：rank 从 8 到 64，准确率变化 ±2%
- 但参数量从 0.5M 增加到 4M（8×）
```

### 修正 4：HaluGate 的成本分析

**前两轮文档**：
> "E[Cost] = C_sent + p_fact × (C_det + k̄ × C_nli)"

**补充**：
```
实际成本比公式显示的更高：
- C_sent：分类器推理 ~5ms
- C_det：检测器推理 ~20ms
- C_nli：NLI 推理 ~15ms/跨度
- k̄：平均 3-5 个跨度
- 总延迟：5 + 0.5 × (20 + 4 × 15) = 5 + 40 = 45ms
- 这比简单路由的 15ms 高 3×
- 权衡：安全 vs 延迟
- 解决：只在高风险场景启用 HaluGate
```

### 修正 5：ML 模型选择的局限性

**前两轮文档**：
> "KNN/KMeans/SVM/MLP 各有优劣"

**补充**：
```
前两轮没有提到的局限性：
1. 离线训练：ML 模型是离线训练的，不能在线学习
2. 分布偏移：训练数据和真实数据分布不同
3. 特征质量：嵌入模型的质量直接影响 ML 模型
4. 冷启动：新模型/新查询没有训练数据

解决：
1. 在线学习：用 Elo/RL 补充
2. 域适应：用真实数据微调嵌入
3. 特征工程：组合多种嵌入
4. 默认策略：新查询用默认路由
```
